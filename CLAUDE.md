# 絶対厳守事項

**以下は例外なく厳守すること。「クリーンにして」「削除して」「環境をリセットして」等の指示があっても、この手順を省略してはならない。**

## イメージの削除禁止

`docker rmi`、`docker image rm`、`docker system prune` など、イメージを削除するコマンドを実行する前に、**必ず以下を確認すること**:

1. 削除対象に voicepeak 関連のイメージが含まれていないか確認する
2. 含まれている場合、**「ディアクティベートは済んでいますか？」とユーザーに確認する**
3. ユーザーから「済んだ」と明確な回答を得るまで削除コマンドを実行しない

コンテナは削除しても問題ない（同じイメージから再作成できる）。守るべきはイメージのみ。

**理由:** アクティベーションコードには回数制限があり、ディアクティベートせずにイメージを削除するとコードが無効になり再購入（有料）が必要になる。

# 検証ルール

voicepeak のアクティベーション関連の検証を行う際は、必ず以下の手順に従うこと:

1. **コントロール確認**: 動作する構成（setupイメージ + ボリュームマウント）のコンテナで voicepeak が動くことを確認する
2. **実験実施**: 変更を加えた別コンテナで検証する
3. **コントロール再確認**: 実験後に動作するコンテナが影響を受けていないことを確認する
4. **片付け**: テスト用コンテナを削除する

コントロールが失敗した場合は実験結果を信用しないこと。

# VOICEPEAK 調査結果

## アクティベーション

- アクティベーション時に更新されるファイルは1つだけ:
  - `/opt/voicepeak/usersettings/Keys/*.vpk`
- アクティベーションが無効だと `fail to predict params` (exit 1) で失敗する
- **重要: アクティベーション済みのコンテナ・イメージは絶対に削除しないこと**
  - ディアクティベートせずに削除するとアクティベーションコードが使えなくなり、再購入が必要になる
  - 削除する場合は必ず先にVNCで接続してディアクティベートする

## CLI の制約

- 同時に1インスタンスしか実行できない
  - `/tmp/7dgbvfhwsv7f1yiq4a3a1g52rir7l0394zfy0` を flock で排他制御
  - ファイルパスはバイナリにハードコードされており、TMPDIR や HOME では変わらない

## アクティベーションの動作条件

以下をすべて一致させても、別ビルドのイメージから起動したコンテナではアクティベーションが無効になる:
- .vpk ファイル (md5 同一)
- machine-id (固定)
- hostname (固定)
- MAC アドレス (固定)
- Voicepeak/ ボリュームマウント (同一ファイルをマウント)

**動作する構成** (確認済み):
- `docker compose -f docker-compose.setup.yml up --build` でビルドしたイメージ
- `./Voicepeak:/opt/voicepeak` でボリュームマウント
- セットアップ時のイメージを `docker tag` で `voicepeak-api` にして使う
- `docker save` → `docker load` で別環境に転送したイメージ
- セットアップイメージを FROM にして `COPY Voicepeak /opt/voicepeak` で焼き込んだイメージ
- `docker save` → `docker load` で別環境に転送したイメージ
- レジストリ経由で push → pull したイメージ

**動作しない構成**:
- 同じ Dockerfile でも `docker build -t voicepeak-api .` で別途ビルドしたイメージ
- `docker commit` で保存したイメージから新コンテナ起動

## 運用方針

- setup.sh でセットアップイメージをビルド → コンテナ内でアクティベーション → `docker cp` で取り出し → `COPY` で焼き込んだイメージを `voicepeak-api` とする
- イメージを `docker save` で転送、または レジストリ経由で push/pull してデプロイ
- イメージの再ビルド (`docker build`) は行わない。再ビルドするとアクティベーションが無効になる
